# 基于 Java 的轻量级关系型数据库 MiniDB MVP 架构设计与底层实现深度研究报告

## 引言与架构上下文深度解析
在当代关系型数据库底层架构的演进历程中，从零开始构建一个具备工业级潜力的系统需要对硬件特性、操作系统内存模型以及并发控制理论有极为深刻的理解。MiniDB 作为一个旨在对标 PostgreSQL、针对 macOS 环境进行深度优化的开源关系型数据库项目，其初始架构决策展现出了显著的前瞻性。项目采用了“控制面与数据面分离”的核心设计理念，其中外层 Client Layer 使用 Spring Boot 构建 RESTful API 以处理网络接入、请求路由与可视化控制，而内部从 Execution Layer 到 Physical Storage Layer 则采用纯 Java（Pure Java）实现，以追求极致的零拷贝（Zero-Copy）与低垃圾回收（Low GC）开销。这种设计不仅在工程实现上隔离了网络 I/O 的复杂性，也使得底层数据引擎能够专注于指令周期的优化与内存吞吐量的提升。
在最小可行性产品（MVP）的边界定义中，MiniDB 被严格限制为仅支持单表的 `SELECT` 和 `INSERT` 操作，并辅以基础的 `WHERE` 条件过滤。计算引擎采用了经典的火山模型（Volcano Model），通过 `SeqScan` 与 `Filter` 算子完成数据的拉取与处理。更为关键的是，MVP 必须在摒弃传统重量级锁管理器（Lock Manager）的前提下，基于多版本并发控制（MVCC）跑通“读已提交”（Read Committed）的基础事务隔离级别。这三大前置条件不仅框定了 MVP 的工程范畴，更对底层架构提出了严苛的挑战。如何在火山模型的流水线中优雅地注入事务可见性判断，如何在 Java 这一运行于虚拟机之上的语言中规避对象分配带来的 Full GC 灾难，以及如何在无锁并发模型下实现读写完全不互斥，构成了本报告深入研究的核心痛点。
本研究报告将通过对数据库底层机制的深入解构，结合学术界的权威理论文献与开源生态中的顶级 Java 数据库实践，为 MiniDB 提供全景式的架构验证与参考实现蓝图。报告将摒弃松散的罗列，以连贯的工程逻辑和严密的系统科学为基础，逐一击破上述三大架构难点。

## 第一章：火山模型与多版本并发控制的深度融合体系
自 Goetz Graefe 于 1994 年提出火山模型（Volcano Iterator Model）以来，按需拉取（Pull-based）的迭代器范式已成为关系型数据库执行引擎的工业标准。然而，当火山模型与多版本并发控制（MVCC）结合时，系统面临着计算流与事务状态流高度耦合的风险。在单表 `SELECT` 查询中，叶子节点算子 `SeqScan` 负责从物理存储（Access Layer）读取原始元组（Tuple），但物理元组本身并不包含当前执行上下文的信息。

### 1.1 执行层与事务层的解耦契约
为了保持架构的纯粹性与可测试性，`SeqScan` 算子绝对不能直接持有事务层的全局状态指针，更不能在其内部实现复杂的事务状态解析逻辑。合理的架构范式是引入一种上下文注入机制。在查询计划（Query Plan）转化为物理执行计划（Physical Plan）并初始化的阶段，执行层（Execution Layer）需要向所有叶子节点算子注入一个轻量级的 `TransactionContext`（事务上下文）对象。
当 `SeqScan` 调用底层存储接口（例如 `HeapFile.readNextTuple()`）时，物理引擎返回的仅仅是一个包含完整元数据头部（Header）的连续字节序列或指针。在 PostgreSQL 的经典实现中，这个头部包含了创建该元组的事务标识符（`xmin`）以及可能删除或更新该元组的事务标识符（`xmax`） 。在 MiniDB 的 MVP 阶段，由于仅支持 `INSERT` 和 `SELECT`，`xmax` 字段可以暂时忽略或预留为特殊标记。`SeqScan` 获取该物理元组后，通过解析出头部的 `xmin` 字段，立即将其传入已注入的 `TransactionContext.isVisible(xmin)` 方法中。只有当该方法返回真值时，`SeqScan` 算子才将该元组作为有效记录返回给上游的 `Filter` 算子进行后续的谓词计算。这种设计不仅将可见性检查的算法复杂度完全封装在事务层内部，还使得算子模型具备了无状态的高效特性。

### 1.2 读已提交（Read Committed）隔离级别的可见性数学推演
实现“读已提交”隔离级别的核心在于，每次执行语句时（在当前 MVP 语境下即每次触发 `SELECT` 请求时），系统必须为当前查询构建一个瞬时的全局活跃事务快照（Snapshot）。这个快照定义了在特定时间点，哪些数据是对当前观察者绝对确定的。
参考现代内存数据库如 DuckDB 与 HyPer 的极简可见性设计机制 ，我们可以建立如下严格的数学评估模型。假设当前正在执行 `SELECT` 的事务标识为 $T_{current}$，系统在查询起始点截取到一个活跃事务集合 $S = \{t_1, t_2, \dots, t_n\}$。同时，系统记录当前已知的最小活跃事务 $T_{min}$（即集合 $S$ 中的最小值）以及系统即将分配的下一个全局事务标识 $T_{max}$（即快照高水位线）。
当 `SeqScan` 截获任意元组并提取其 `xmin` 时，可见性判定遵循严格的短路求值（Short-circuit evaluation）逻辑：首先，如果 `xmin` 恰好等于 $T_{current}$，说明该元组是本事务在之前操作中刚刚插入的，根据 ACID 的要求，这种“读己之所写”必须始终可见。其次，如果 `xmin` 大于或等于 $T_{max}$，说明创建该元组的并发 `INSERT` 事务是在当前 `SELECT` 语句开启快照之后才启动的。由于这些元组属于未来状态，对当前事务必定绝对不可见 。
最核心的判断落在历史时间线内。如果 `xmin` 严格小于 $T_{min}$，这表明创建它的事务在当前快照生成前已经彻底结束（提交或回滚）。此时，判定逻辑无需查阅复杂的活跃集合，只需直接访问轻量级的全局提交日志（Commit Log，简称 CLOG），只要其状态标记为 `COMMITTED`，则数据无条件可见 。而当 `xmin` 处于活跃窗口内部，即满足 $T_{min} \le xmin < T_{max}$ 时，判定逻辑需要查询活跃集合 $S$。如果 `xmin` 存在于 $S$ 中，说明该创建事务虽然发生得早，但至今尚未完成提交，因此不可见；反之，若 `xmin` 不在 $S$ 中且在 CLOG 中标记为已提交，则数据可见 。

| 可见性条件分支 | 数学关系表达 | 事务状态解释 | 判定结果 |
| --- | --- | --- | --- |
| 自身事务 | xmin==Tcurrent​ | 当前事务自行产生的数据 | 可见 |
| 未来事务 | xmin≥Tmax​ | 在快照截取之后启动的事务产生的数据 | 不可见 |
| 历史终结事务 | xmin<Tmin​ | 绝对发生于快照之前的事务（需检查 CLOG 确认提交） | 可见（若已提交） |
| 活跃窗口事务 (未提交) | Tmin​≤xmin<Tmax​ 且 xmin∈S | 事务启动于快照之前，但快照截取时尚未结束 | 不可见 |
| 活跃窗口事务 (已提交) | Tmin​≤xmin<Tmax​ 且 xmin∈/S | 事务处于快照区间，但在快照截取前已碰巧完成提交 | 可见（若已提交） |
通过上述矩阵模型，可见性检查被退化为一系列极低成本的标量比较和位运算。这种设计极大地减少了在千万级数据扫描时对 CPU 缓存（L1/L2 Cache）的污染，使得 Volcano 模型的拉取操作在时间复杂度上趋近于纯粹的内存拷贝。

## 第二章：Java 内存环境下的零拷贝（Zero-GC）版本链机制
在采用 C++ 或 Rust 编写的数据库系统中，内存的分配与释放由开发者精确控制。然而，在 Java 虚拟机（JVM）生态中，任何对象的创建都会在堆（Heap）上分配内存。如果在 MiniDB 的底层架构中，为每一行从磁盘读取的记录或每一个插入操作分配一个具体的 `Tuple` 对象，甚至为版本链分配由对象引用构成的链表节点，系统在遭遇高并发负载时将不可避免地触发内存溢出或极为频繁的 Stop-The-World（STW）全局垃圾回收暂停 。

### 2.1 堆外内存模型与内存映射的架构演进
为了彻底解决 JVM 的垃圾回收痛点，MiniDB 必须抛弃面向对象编程范式在数据通道上的诱惑，转而采用面向数据的内存架构（Data-Oriented Design）。在 Java 中，规避 GC 的最成熟方案是大量运用堆外内存（Off-Heap Memory）与操作系统的内存映射机制（Memory-Mapped Files）。
底层的数据访问层（Access Layer）应当负责管理一个巨大的连续地址空间。通过 Java 的 `java.nio.channels.FileChannel.map` 或更为底层的 `sun.misc.Unsafe` API，系统能够直接将磁盘上的物理文件映射到内存页面缓存（Page Cache）中 。在此架构下，一个 8KB 的数据页（Data Page）在内存中仅表现为一个原生字节缓冲区。当执行引擎需要处理某一行记录时，它不再接收一个被实例化的 Java 对象，而是接收一个被称为“游标”（Cursor）或“元组指针”（Tuple Pointer）的极轻量级数值结构，通常由一个 64 位的长整型（Long）构成：前 32 位表示页面 ID，后 32 位表示该元组在页内的字节偏移量。
所有针对记录字段的操作（如读取整数、提取字符串）必须被推迟到最后一刻（Late Materialization）。算子通过类似享元模式（Flyweight Pattern）的设计，将这枚指针指向内存块的特定偏移量，利用 `Unsafe.getInt()` 等原生方法直接从操作系统的内存中读取比特流 。这种零拷贝机制不仅将 Java 对象的创建率降至绝对的零，还极大程度地贴合了现代 CPU 的缓存预取（Cache Prefetching）机制，实现了性能的飞跃。

### 2.2 契合 MVP 阶段的版本存储（Version Storage）策略
在 MVCC 的理论体系中，版本存储的物理布局通常被划分为三种主流范式：增量存储（Delta Storage）、时间旅行存储（Time-Travel Storage）以及仅追加存储（Append-Only Storage）。不同存储范式对内存碎片、缓存局部性以及 GC 压力有着截然不同的影响。

| 版本存储策略 | 核心机制描述 | MVP 契合度与工程成本评估 | 针对 Java GC 的影响 |
| --- | --- | --- | --- |
| Delta Storage (增量) | 主表保存最新数据，Undo 空间保存逆向重做字段。 | 低契合度：逻辑极其复杂，需要频繁的指针解引用，适合更新密集型场景。 | 指针跳转多，若实现不当易产生零散对象，对 Java 堆不友好。 |
| Time-Travel (时间旅行) | 主表保存最新数据，旧版本完整复制到独立的时间轴表中。 | 中等契合度：适合历史查询，但存在空间双倍消耗。 | 需管理两个不同结构的内存池，内存回收机制相对复杂。 |
| Append-Only (仅追加) | 不区分主表与历史表，新版本直接作为新记录追加到同一个连续的物理页空间末尾。 | 极高契合度：MVP 仅涉及 INSERT，天然就是追加写，完全无需处理逆向修改。 | 可使用巨大的连续直接内存缓冲区，完全消除碎片化与对象分配。 |
针对 MiniDB 仅要求 `INSERT` 和并发 `SELECT` 的 MVP 边界，最为完美的选择是 **Append-Only Storage**（仅追加存储）。由于不存在 `UPDATE` 操作，自然无需维护复杂的反向 Undo 日志链条。每一次 `INSERT` 操作仅仅是在预先分配好的堆外连续内存池中，通过原子加法（Atomic Addition）申请一段连续的字节区间，并将数据顺序写入。对于并发的 `SELECT` 而言，它只是在遍历这片内存区域的指针。
为了给未来的版本迭代（例如后续支持 `UPDATE` 和 `DELETE`）预留演进空间，MiniDB 可以设计一个扁平化的高效 Undo Buffer 数组机制。每个处理连接或事务都可以持有一个线程局部（Thread-Local）的直接内存块，作为其专属的日志写入区 。Java 内存中的版本链管理无需使用 `NextNode` 这样的对象引用，而是完全采用原生数据类型构建虚拟指针。这种结构极度紧凑且完全不受 JVM 垃圾收集器的监控。

### 2.3 基于原生位图的快照状态管理
管理事务可见性的难点在于如何快速查询 `xmin` 是否存在于活跃事务集合中。传统的 Java 开发模式可能会倾向于使用 `java.util.HashSet<Long>`，但这将带来致命的性能开销，因为频繁的自动装箱（Autoboxing）操作会生成数以万计的 `Long` 对象对象，迅速填满新生代（Young Generation）内存。
为了避免这种高昂的代价，业内最佳实践是利用环形缓冲区（Ring Buffer）配合定制化的位图（BitSet）来表示事务集合 。在底层，可以维护一个长整型数组（`long`），每个位（Bit）代表一个事务 ID 的状态。由于事务 ID 在局部时间窗口内是单调递增的，可以通过将事务 ID 减去当前窗口基准值，将其映射到数组的具体位置上。在进行快照拷贝或可见性检查时，位运算指令（Bitwise Operations）可以在单条 CPU 指令周期内判定某一位的布尔状态，这种方式不仅内存消耗极少，更实现了绝对意义上的无 GC 运行 。

## 第三章：无锁化设计下的极简并发控制与读写分离
在成熟的数据库系统中，为了维护严格的事务串行化（Serializability），通常会引入复杂的锁管理器（Lock Manager），采用严格的两阶段锁（Strict 2PL）机制在表级或行级强制排他访问 。然而，在 MiniDB 的 MVP 阶段，实现一个笨重的锁管理器不仅是对计算资源的巨大浪费，更容易因锁竞争和死锁检测而导致系统吞吐量断崖式下跌 。如何在没有任何重量级锁的情况下实现 `INSERT` 与 `SELECT` 操作互不阻塞，是检验架构师功底的试金石。

### 3.1 物理空间的原子化分配机制
在只有 `INSERT` 操作的并发环境中，写冲突的本质仅仅是多个线程试图同时写入物理文件的同一块尾部区域。要解决这一问题，完全无需引入操作系统级别的互斥锁（Mutex）或 Java 的 `synchronized` 关键字。
我们可以在每一个正在接受写入的内存页（Page）头部维护一个无锁的偏移量指示器，采用 Java 的 `java.util.concurrent.atomic.AtomicInteger` 或者利用 `Unsafe` 的硬件级别比较并交换指令（Compare-And-Swap, CAS）操作 。当事务准备写入一条长度为 $L$ 字节的记录时，它只需执行一次 `fetchAndAdd(L)` 的操作去获取自己的写入区间。如果 CAS 成功，该线程就获得了绝对排他、独立的一段内存切片，随后可以从容不迫地将数据序列化进去；如果失败，则仅需要短暂地重试（Spin-wait）。这种原子性的游标推进技术（Atomic Bump Pointer）能够支撑每秒数百万次的高频并发插入，且各线程的内存写入操作物理隔离，根本不产生逻辑数据冲突。

### 3.2 提交日志（CLOG）的全局无锁状态机
并发控制的另一核心是事务状态转移的原子性。当一个写事务完成内存拷贝后，它必须向系统宣告其产生的元组已处于全局可见状态。正如早期的 PostgreSQL 架构中设计的提交日志机制 ，MiniDB 也可以在内存中建立一个高度紧凑的全局状态数组（Commit Log Array）。
在没有 `UPDATE` 回滚需求的简单追加写场景中，事务的状态机只有三种节点：进行中（IN_PROGRESS）、已提交（COMMITTED）和已回滚（ABORTED）。因此，每个事务的当前状态仅仅需要占用 2 个比特（Bit）的存储空间。一个原生的 32 位整型（`int`）可以精确地打包 16 个相邻事务的状态，而一个几十兆字节的原生数组足以追踪上亿个事务的历史状态 。
在写事务最后执行 `COMMIT` 指令时，只需通过原子位掩码更新（Atomic Bitwise Update）将其在 CLOG 数组中对应的两个比特翻转为已提交状态，并视情况推进系统的全局最老活跃事务指针。整个提交过程没有获取任何排他锁，其操作的原子性完全由现代多核处理器的缓存一致性协议（MESI）与内存屏障（Memory Barriers）来保障。

### 3.3 读不阻塞写、写不阻塞读的绝对隔离
基于上述无锁化页面分配与原子化状态标记，MiniDB 在宏观上实现了完美的并发隔离。当读取操作（`SELECT` 事务）启动时，它在纳秒级的时间内拷贝一份当前 CLOG 状态指针，形成自己的不可变可见性快照（Read View）。
在此之后，无论后台有多少并行的 `INSERT` 事务在同一张表上原子推进尾部指针并写入新数据，读取操作的遍历逻辑只需遵循严格的数学判定（见本文 1.2 节）：遇到新写入的元组由于其 `xmin` 不满足快照条件将被直接跨过（Skip）。这种基于数据纪元（Epoch-based）的隔离策略，让系统彻底摆脱了读写互斥锁的束缚，实现了最高级别的系统吞吐能力 。

## 第四章：架构设计的权威学术论证与文献映射
数据库的底层架构是一门严谨的系统工程科学，所有关于并发控制与内存管理的工程直觉都必须建立在坚实的学术验证基础之上。本节提取了三篇在全球数据库顶会上发表的具有里程碑意义的经典文献，详细剖析它们对 MiniDB 架构的支撑逻辑。

### 4.1 主存数据库中高性能串行化多版本并发控制研究 (HyPer 模型)

- **文献标题**：Fast Serializable Multi-Version Concurrency Control for Main-Memory Database Systems 
- **作者团队**：Jan Böttcher, Viktor Leis, Thomas Neumann 等（慕尼黑工业大学 HyPer 数据库开发团队）
- **摘要深度解析**：传统的多版本系统往往为了维护历史版本而牺牲了主线数据的扫描性能，同时为了保证串行化级别引入了高昂的并发控制成本。该论文提出了一套革命性的主存数据库 MVCC 实现机制：所有数据均采用原地更新（In-place update）以最大化扫描速度，而不可见的历史快照则以后向差量（Before-image deltas）的形式存储在 Undo 缓冲区中。研究通过将多版本验证机制与精准锁定（Precision locking）结合，以极低的成本实现了完美的事务隔离。
- **架构映射与指导意义**：对于 MiniDB 而言，HyPer 的设计哲学证明了“保持数据在物理块上连续存放”对现代 CPU 缓存机制的巨大优势。该文强烈支持了本报告第二章提出的尽量利用直接内存进行连续字节存储的策略，并指出事务历史应当仅作为旁路记录存在，绝不能干扰主线数据的高速遍历 。

### 4.2 内存多版本并发控制机制的实证评估分析 (VLDB 2017 经典评测)

- **文献标题**：An Empirical Evaluation of In-Memory Multi-Version Concurrency Control 
- **作者团队**：Yingjun Wu, Andrew Pavlo 等（卡内基梅隆大学与新加坡国立大学联合团队）
- **摘要深度解析**：作为数据库体系结构领域的奠基性文献之一，本文首创性地将复杂的 MVCC 架构解耦为四个独立的正交设计维度：并发控制协议（Concurrency Control Protocol）、版本物理存储（Version Storage）、内存垃圾回收（Garbage Collection）与多版本索引管理（Index Management）。作者在同一实验框架内对不同组合进行了海量负载压力测试，系统地揭示了在高并发写读场景下各个协议的性能上限与硬件瓶颈。
- **架构映射与指导意义**：该论文在关于“版本物理存储”维度的实验结论，为 MiniDB 在 MVP 阶段选用仅追加（Append-Only）存储范式提供了强有力的数据背书。其实验明确指出，在缺乏回滚需求的极简读写负载下，Append-Only 在消除系统锁竞争与抑制碎片产生方面具有压倒性优势，这一结论是本报告第三章无锁并发设计的理论基石 。

### 4.3 vMVCC：可形式化验证的高性能事务库架构研究 (OSDI 2023 顶会成果)

- **文献标题**：Verifying vMVCC, a high-performance transaction library using multi-version concurrency control 
- **作者团队**：Yun-Sheng Chang, Ralf Jung 等（麻省理工学院等）
- **摘要深度解析**：发表于计算机系统领域最高级别会议 OSDI，探讨了如何在保证内存操作与事务可见性绝对正确的前提下，构建极速的 MVCC 底层库。vMVCC 的亮点在于，它是全球首个带有机器检验正确性证明（Machine-checked proof）的实现。为了解决并发瓶颈，研究团队利用了底层硬件的时间戳计数器（RDTSC）替代了易引发总线风暴的传统原子递增变量。
- **架构映射与指导意义**：虽然 MiniDB 在 MVP 阶段对正确性的要求高于极限性能，但 vMVCC 中处理并发时间戳分配与预防幽灵异常的思路极具启发性。它警告我们在 Java 实现中，即使是 `AtomicLong.incrementAndGet()` 也可能在高压下因缓存行伪共享（False Sharing）而成为瓶颈。这提示 MiniDB 在分配事务 ID 阶段可以引入批量获取策略，以规避全局锁的性能塌陷 。

## 第五章：开源生态洞察与高质量参考实现对比
工业界与学术界的顶级开源项目能够为架构规划提供极为宝贵的实现模板。通过深入剖析这些项目在并发处理与内存组织上的代码模式，可以为 MiniDB 梳理出明晰的演进路线。

### 5.1 教育级标准实现：MIT 6.830 / CMU 15-445 (SimpleDB)
作为麻省理工学院与卡内基梅隆大学的经典数据库内核教学项目，SimpleDB 及其诸多 Java 重构版在结构设计上具有非凡的清晰度 。

- **代码组织范式**：SimpleDB 极好地实现了组件边界的切割。其核心拥有一个单例的 `BufferPool` 控制所有页面出入，物理数据读取被封装于 `HeapFile`，而操作执行则完美体现了 Volcano 模型，每一层都继承了迭代器接口（如 `DbIterator`）进行 `open()` 和 `next()` 的嵌套调用 。
- **架构批判与反思**：在其原始版本中，为了实现隔离级别，SimpleDB 在 `BufferPool` 层面注入了一个极为厚重的锁管理器（Lock Manager），采用严格两阶段锁来阻止并发读写冲突 。一旦某个事务请求读取某页，整个物理页就会被加上共享锁，直接阻塞了写入事务。
- **移植建议**：MiniDB 应当全盘吸收 SimpleDB 在 Volcano 执行链与物理存储解耦上的优秀代码框架设计，但是必须在 `DbIterator.next()` 层面上彻底移除其陈旧的 `LockManager` 逻辑。取而代之的，是将上文论证的基于快照与位图运算的 `VisibilityContext` 嵌入其元组抓取逻辑之中。

### 5.2 零拷贝极致工业标杆：QuestDB
QuestDB 是一个专门为金融级吞吐量设计的时序与关系型数据库，采用 Java 语言编写，其最大的技术标签即是 Zero-GC（零垃圾回收） 。

- **代码组织范式**：在代码层面上，QuestDB 几乎重写了整个 Java 基础结构，摒弃了标准库中的对象模型。其核心模块使用 C++ 和 Rust 编写底层极速 IO 操作，然后通过 Java Native Interface（JNI）与其定制的 Java 代码无缝连接。在其 Java 代码库中，大量出现了通过直接内存地址偏移进行数值操作的逻辑，彻底取代了对象图遍历 。
- **架构批判与反思**：这种深度的操作系统级优化带来了令人震撼的写入性能（单机百万行/秒），但也使得代码丧失了大部分跨平台编译和阅读的便利性，过于接近汇编层次。
- **移植建议**：MiniDB 作为一个追求可读性的开源项目，不能盲目复制 QuestDB 的 JNI 集成模式。然而，QuestDB 利用内存映射文件（MMap）实现持久化存储以及利用堆外内存避免 Java 垃圾回收器干扰的思想（如上文第二章所述）是 MVP 阶段构建高性能缓冲池的必修课 。

### 5.3 工业级轻量嵌入引擎：H2 Database (MVStore 机制)
H2 作为全球使用最广泛的轻量级 Java 嵌入式关系型数据库，不仅支持极其完整的 SQL 方言，其内核演进同样发人深省 。

- **代码组织范式**：H2 放弃了老旧的基于页锁定的 PageStore，全面拥抱了基于无锁化设计的 MVStore（多版本存储机制）。MVStore 采用了类似日志结构合并树（LSM-Tree）的持久化理念，将所有的变更以后追加的形式写入磁盘，同时在内存中维护了一套基于多版本数据结构 B-Tree 的映射层，天然支持了高并发下的读写分离隔离 。
- **架构批判与反思**：尽管 H2 的 MVCC 处理得很稳健，但它由于过度依赖 Java 自身的垃圾回收机制来清理不可见的历史节点，在承载极度密集的事务压力时，偶尔仍会出现内存回收滞后导致的性能毛刺（Spike）现象 。
- **移植建议**：H2 的代码证明了纯 Java 实现企业级 MVCC 是绝对可行的。MiniDB 应借鉴其不设传统读锁的策略，同时通过引入自研的原生内存池（Arena Allocator）或者定长的位图状态机来规避 H2 面临的垃圾回收延迟痛点，从一开始就切断对象分配的源头。

| 评估维度 | MIT SimpleDB (教育级) | QuestDB (极致时序) | H2 MVStore (轻量级) | MiniDB MVP 演进建议策略 |
| --- | --- | --- | --- | --- |
| 隔离与并发 | 严格两阶段锁（2PL），读写互斥严重阻塞 | 完全无锁追加，硬件级时间戳隔离 | 无读锁 MVCC，事务行级写锁等待 | 摒弃 2PL，基于全局轻量级 CLOG 快照实现完全无锁 |
| 内存与 GC | 对象密集型（Object-heavy），频繁触发 GC | 完全利用堆外内存与 JNI，严格 Zero-GC | 依赖 JVM 内存管理机制进行旧版本 GC | 采用 Unsafe 或 ByteBuffer 堆外映射，仅在输出端物化对象 |
| 物理存储 | 简单的堆文件（HeapFile）离散读写 | MMap 内存直接映射，按时间跨度分区 | 类似 LSM 机制的日志结构 B-Tree | MVP 期采用 MMap 连续内存映射文件的 Append-Only 方式 |

### 时序核心原理与生命周期解析

1. **全局状态无锁冻结（步骤 2-3）**：在 REST 请求触达执行器入口之前，控制面首先与底层的并发控制层对话。此时系统以极其轻量级的方式读取当前 CLOG 的元数据信息，生成当前事务的 `TransactionContext`。这一刻，系统的活跃事务快照（Active Tx BitMap）被冻结并嵌入到上下文中，确保无论后续执行持续多长时间，该查询对系统数据的观测视角被严格锁定。
2. **指针传递与物化延迟（步骤 6-8）**：当拉取引擎触发下层存储动作时，Access Layer 操作的是操作系统底层的 MMap 内存区域。引擎读取记录内容时绝不触发任何 `new Object()` 操作，仅仅是依靠原生的整数值游标定位。这种方法最大程度地绕过了 JVM 的屏障，提升了流水线吞吐。
3. **内联的短路判定机制（步骤 9-11）**：此时架构进入解耦机制的最高潮。`SeqScan` 不需要了解复杂的事务管理细节，仅仅是将物理块前置字节中解析出的整数 `xmin` 委托给 `TransactionContext`。这使得底层的算子仅仅充当纯粹的数据搬运工。
4. **按需实例化对象（步骤 13-14）**：在数据过滤完成、可见性均验证通过后，在向 Spring Boot 边界返回的最后一刻，那些最终需要展示给用户的极少部分数据，才通过底层的缓冲区块按需解包为常规的 Java 实体类对象并完成 JSON 序列化。这样，大部分不可见或不满足 `WHERE` 过滤的废弃元组在内存中就如同从未存在过一般，实现了极低 GC 的设计初衷。

## 结语与演进展望
在自主构建工业级关系型数据库的漫长征途中，MVP 阶段底层架构设计的精准度往往决定了整个产品生命周期的技术上限。本研究报告通过对火山执行模型与事务可见性检查之间边界的重新划分，为 MiniDB 提供了一套清晰且高内聚的解耦契约。通过深入拆解 JVM 在数据库系统中的天然软肋，报告创造性地指出了利用直接内存与原生游标数组规避系统级垃圾回收停顿的工程路线。而在并发控制领域，基于多版本状态数组（CLOG）和位掩码运算的极简追加写策略，更是为系统在单表高并发场景下实现了读写隔离提供了无锁化的核心解法。
这三大架构策略构成了支撑起 MiniDB MVP 产品的铁三角。在这个坚固的基础上，系统不仅能够游刃有余地通过初期的基准测试挑战，而且为未来向多表 Join 关联、B+树复杂索引引入、以及序列化（Serializable）等高级隔离级别的功能拓展，保留了极度宽广的理论空间与弹性设计余地。随着项目的逐步开源与迭代，这套高度定制化的 Java 引擎必将在低延迟事务处理领域展现出非凡的架构魅力与应用潜力。
